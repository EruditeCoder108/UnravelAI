Diagnosis

Observed Symptom

The UI does not update after adding, moving, or deleting tasks. Tasks are not rendered in their respective columns, and their status changes are not reflected visually. However, the 'Reset Board' button correctly clears the display, and log messages confirm state changes are processed.

Reproduction Path

  1. 1. Open `index.html` in your browser.
  2. 2. Add a new task using the 'Add Task' button (e.g., "Buy milk", Medium priority).
  3. 3. Observe that the newly added task does *not* appear in the 'To Do' column on the board.
  4. 4. Click the 'Reset Board' button. The board will correctly clear all tasks.
  5. 5. Add another task. It will still not appear on the board, demonstrating the cache issue persists across individual task modifications.

Confidence Evidence

VERIFIED:
  • The `select` function in `store.js` (L12-L13) checks `if (cached && cached.input === state.tasks)`. This condition only becomes false if the `state.tasks` array reference itself changes, not when its contents are modified.
  • The `addTask` (store.js L24), `moveTask` (store.js L31), and `deleteTask` (store.js L38) functions modify the `state.tasks` array in place using `push`, property assignment (`task.status = newStatus`), or `splice`. These operations do not change the `state.tasks` array's memory address.
  • The `resetBoard` function (store.js L43) explicitly reassigns `state.tasks = []`, which creates a new array reference, and also correctly calls `selectorCache.clear()` (store.js L45). This is why 'Reset Board' correctly updates the UI and clears the cache.

Concept To Learn

Data Flow / Caching Logic

Cache Invalidation for Mutable Data

Caches are used to make apps faster by remembering results. But if the original data changes and the cache isn't told to update or forget (invalidate), it will keep showing old information. This is like a shopkeeper having old prices written on a board after the actual price of an item has changed in the inventory. Customers will pay the wrong (old) price.

PATTERN TO AVOID:

Relying on object reference equality (`===`) for cache invalidation when the object's *contents* are mutated in place. If data inside a container changes, but the container itself (its memory address) stays the same, a cache that only checks the container's address will think nothing has changed and give you old data.

💡 Imagine you have a 'to-do' list written on a whiteboard (`state.tasks`). Your friend, who updates the board, keeps a small sticky note in their pocket with the count of 'High Priority' tasks (`selectorCache`). When you add a new 'High Priority' task to the whiteboard, your friend updates the whiteboard directly. But they forget to update the sticky note in their pocket. So, next time someone asks, 'How many High Priority tasks?' your friend looks at their sticky note and gives the old, wrong count, even though the whiteboard has the correct, new task. The sticky note (cache) needs to be thrown away or updated (`selectorCache.clear()`) every time the whiteboard (mutable `state.tasks`) changes, no matter how small the change.

Invariant Violations

  • The `state.tasks` array must always represent the current, authoritative list of tasks.
  • The `selectorCache` must always be cleared whenever the *contents* of `state.tasks` are modified by `addTask`, `moveTask`, `deleteTask`, or `resetBoard` to ensure `select` provides fresh data.
  • Every state-modifying action (add, move, delete, reset) must trigger a `scheduleUpdate` followed by `emit('state:change', getSnapshot())` to ensure UI updates are batched and dispatched.
  • The `id` property of a task must be unique across all tasks.
  • Task `status` must be one of `todo`, `doing`, or `done`.
  • Task `priority` must be one of `high`, `medium`, or `low`.

Technical Root Cause

The `select` function's cache (`selectorCache`) in `store.js` uses strict reference equality (`cached.input === state.tasks`) to determine if cached data is still valid. However, when tasks are added, moved, or deleted, the `state.tasks` array is modified *in place* (e.g., using `push`, `splice`, or changing properties of objects within the array). Because the array object's reference itself does not change, the `select` function mistakenly believes the data is unchanged and returns stale, cached results to the UI rendering functions.

Minimal Code Fix

File: bugcode8/store.js L12
Modify the `addTask`, `moveTask`, and `deleteTask` functions in `bugcode8/store.js` to clear the `selectorCache` after modifying `state.tasks`. 

```javascript
// bugcode8/store.js

export function addTask(name, priority) {
    const task = { id: state.nextId++, name, priority, status: 'todo' };
    state.tasks.push(task);
    selectorCache.clear(); // <-- ADD THIS LINE
    scheduleUpdate(() => emit('state:change', getSnapshot()));
}

export function moveTask(id, newStatus) {
    const task = state.tasks.find(t => t.id === id);
    if (!task) return;
    task.status = newStatus;
    selectorCache.clear(); // <-- ADD THIS LINE
    scheduleUpdate(() => emit('state:change', getSnapshot()));
}

export function deleteTask(id) {
    const idx = state.tasks.findIndex(t => t.id === id);
    if (idx === -1) return;
    state.tasks.splice(idx, 1);
    selectorCache.clear(); // <-- ADD THIS LINE
    scheduleUpdate(() => emit('state:change', getSnapshot()));
}

// The resetBoard function already clears the cache at L45, which is correct.
```

Why This Works

The problem was that the `select` function's cache, `selectorCache`, was designed to remember previous calculations based on the `state.tasks` array. It only recalculated if the *whole array itself was replaced* (like a new box), not if its *contents changed* (like items inside the same box). When you add, move, or delete a task, the `state.tasks` array's items change, but the array 'box' remains the same. Because the `select` function thought the 'box' hadn't changed (`cached.input === state.tasks` was still true), it would serve up old task lists from its memory. By adding `selectorCache.clear()` after every change to the tasks list, we are telling the system, 'Hey, the tasks have changed, forget everything you thought you knew!' This forces `select` to re-run its filtering logic with the fresh, updated `state.tasks` array, ensuring the UI always gets the latest information and renders correctly.
