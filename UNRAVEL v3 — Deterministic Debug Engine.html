Concept To Learn

STATE_MUTATION

State Synchronization between In-Memory Caches and Persistent Storage.

This bug creates a momentary but critical period of incorrect data (a "stale read"). In authentication, this means a user might successfully log in (receive a token) but the application immediately thinks they are *not* logged in, leading to infinite re-prompts, silent failures, or a broken user experience. This type of bug is especially tricky because the state *eventually* becomes correct, masking the initial failure.

PATTERN TO AVOID:

Asynchronous write-through with synchronous read-through, without immediate in-memory update. When you write new data to a persistent store, and that store then asynchronously notifies other parts of the system about the change, don't rely on the *persistent store's notification* for the immediate consistency of your *in-memory cache*. If a component writes new data to storage, and then needs to immediately read that *new* data, it must update its *own* in-memory copy synchronously, *before* or *during* the write operation. Waiting for the asynchronous notification will lead to stale reads.

💡 Imagine you are a shopkeeper (the `ScopedAccountAccess` class) and you have a small notebook (your `this.value` array) where you write down which customers (accounts) are allowed credit. You also have a big register book (the `SecretStorage`) which is the official record. When a new customer is approved for credit, you write their name in the big register book (`_accountAccessSecretStorage.store`), but you forget to *immediately* write it in your small notebook. Another shop assistant (the `CachedPublicClientApplication._update` method) comes to check if a customer is allowed credit, they look only at your small notebook. Since it's not updated, they say "no credit," even though the customer is now officially approved in the big register book. Only later, when someone else asks for a list of *all* approved customers and you refresh your small notebook from the big register book (`update()` method), does your small notebook become correct.

  • ScopedAccountAccess.value Consistency: The `this.value` array within `ScopedAccountAccess` MUST always accurately reflect the current set of allowed account IDs that are stored in the underlying `SecretStorage` for its specific `_key`. Any direct modification to the allowed accounts list that is persisted MUST also synchronously update `this.value`.
  • Immediate Post-Authentication State: Immediately after a successful interactive authentication and `ScopedAccountAccess.setAllowedAccess(account, true)` is called, the `CachedPublicClientApplication` MUST be able to correctly identify the newly authenticated account as allowed through its `_accountAccess.isAllowedAccess()` check.
  • SecretStorage Eventual Consistency: The `SecretStorage.onDidChange` event, followed by `AccountAccessSecretStorage.onDidChange` and `ScopedAccountAccess.update()`, must always eventually reconcile the `ScopedAccountAccess.value` with the actual `SecretStorage` content.
  • `acquireTokenInteractive` Success: When `acquireTokenInteractive` successfully obtains an `AuthenticationResult`, the associated `result.account` must be considered "allowed" by the system for immediate subsequent operations, if `setAllowedAccess(result.account, true)` was invoked.

  • Uniqueness of homeAccountId: Each `AccountInfo.homeAccountId` must uniquely identify an account within the `ScopedAccountAccess` system to ensure correct filtering and storage.

Technical Root Cause

The root cause is a STATE_MUTATION issue where the in-memory state of `ScopedAccountAccess.value` is not updated synchronously when `setAllowedAccess` is called. This creates a temporary inconsistency between the in-memory cache and the `SecretStorage`, leading to a race condition. When `CachedPublicClientApplication._update` runs immediately after `setAllowedAccess`, it incorrectly filters out the newly authenticated account because `ScopedAccountAccess.isAllowedAccess` relies on the outdated in-memory `value`.

Alternative Hypotheses

  • Hypothesis 1 (STATE_MUTATION - Stale In-Memory State): The `ScopedAccountAccess.value` array, which holds the in-memory list of allowed accounts, is not updated immediately after `setAllowedAccess` is called. Instead, `setAllowedAccess` only writes the *new* state to `SecretStorage` asynchronously. `CachedPublicClientApplication._update`, which runs immediately afterward, reads from the stale `ScopedAccountAccess.value`, causing the newly allowed account to be filtered out prematurely.
  • Hypothesis 2 (ASYNC_ORDERING - Event Processing Delay): The `onDidChange` event from `SecretStorage` that `ScopedAccountAccess` listens to is inherently asynchronous (microtask/event loop based). Even if `setAllowedAccess` was designed to be perfectly synchronous with the `SecretStorage.store` call, the subsequent `ScopedAccountAccess.update` (which relies on `onDidChange` and `SecretStorage.get`) will always run *after* the `CachedPublicClientApplication._update` that filters accounts, leading to a race condition where the account list is processed before it's truly updated.
  • Hypothesis 3 (UI_LOGIC - Incorrect Filtering Logic): The filtering logic itself in `CachedPublicClientApplication._update` (specifically `after.filter(a => this._accountAccess.isAllowedAccess(a))`) is fundamentally flawed for interactive flows. It assumes `isAllowedAccess` will *always* reflect the most current state immediately after `setAllowedAccess`, which might not be true if there are any intermediate asynchronous steps or UI interactions involved. This might indicate an architectural mismatch between how accounts are managed and how they are consumed post-login.
