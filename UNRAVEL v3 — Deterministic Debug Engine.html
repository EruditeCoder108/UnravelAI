Concept To Learn

STATE_MUTATION

Immutability and Reference Equality for State Management and Caching

In modern web applications, especially with tools that rely on reactive updates or memoization (like React's 'useMemo' or Redux's selectors), changes are often detected by comparing references. If you mutate data directly, these change detection mechanisms will fail, leading to parts of your user interface not updating, or showing old data, even if your underlying data *has* changed. Adopting an 'immutable' approach (always creating new objects/arrays when data changes) makes your change detection reliable and your state predictable.

PATTERN TO AVOID:

Directly mutating (changing in place) arrays or objects that are used as cache keys or for change detection based on reference equality (=== comparison).

💡 Imagine you have a special box (your 'state.tasks' array). When you want to check if the contents of the box have changed, you look inside. But if you're a bit lazy, you might just look at the *label* on the box. If the label is the same, you assume the contents are the same. Now, if someone opens the box, adds or removes items, and then closes it *without changing the label*, your lazy check (the '===' comparison) will tell you the box hasn't changed, even though it has! To fix this, every time you change the contents, you should put them in a *brand new box with a new label*. That way, your lazy check (comparing labels) will always work.

Invariant Violations

  • State Consistency: The `state.tasks` array must always reflect the single source of truth for all tasks in the system.
  • Event-Driven Updates: All modifications to `state.tasks` must trigger a `state:change` event through `scheduleUpdate` to ensure UI and other components are notified.
  • Cache Invalidation Integrity: The `select` function's cache must accurately reflect the current `state.tasks`. This means if `state.tasks` content changes, its reference must also change, or a more sophisticated deep comparison or versioning system must be in place.
  • Unique Task IDs: Each task must have a unique `id` to allow for correct identification and manipulation (move, delete). `state.nextId` ensures this for new tasks.
  • DOM Element Existence: All `document.getElementById` calls must target existing elements in `index.html` to prevent runtime errors.
  • Valid Status/Priority: Task `status` and `priority` values should always be from a predefined set ('todo', 'doing', 'done' for status; 'high', 'medium', 'low' for priority) to ensure correct rendering and filtering.
  • Global Function Accessibility: `window._move` and `window._delete` must be defined globally before `renderColumn` is called, as `onclick` attributes directly reference them.
  • Microtask Scheduling Integrity: `scheduleUpdate` must ensure that all queued updates are eventually flushed exactly once per microtask tick, and that `pendingFlush.scheduled` correctly prevents redundant `queueMicrotask` calls.

Technical Root Cause

The core issue is that the `select` function, which is responsible for providing filtered task lists for rendering, uses a caching mechanism that relies on reference equality (`cached.input === state.tasks`) to determine if its cached results are still valid. However, the `addTask`, `moveTask`, and `deleteTask` functions directly mutate the `state.tasks` array in place (e.g., using `push`, `splice`, or modifying task objects within the array). When an array is mutated in place, its memory address (reference) remains the same. This means `cached.input === state.tasks` will incorrectly evaluate to `true` even after the array's contents have changed, causing `select` to return stale, cached data instead of recomputing with the latest tasks. The UI then renders based on this outdated information.

```javascript
// bugcode8/store.js
import { scheduleUpdate } from './scheduler.js';
import { emit } from './event-bus.js';

const state = {
    tasks: [],
    nextId: 1,
};

const selectorCache = new Map();

export function select(selectorFn) {
    const key = selectorFn.toString();
    const cached = selectorCache.get(key);
    if (cached && cached.input === state.tasks) {
        return cached.output;
    }
    const result = selectorFn(state.tasks);
    selectorCache.set(key, { input: state.tasks, output: result });
    return result;
}

export function addTask(name, priority) {
    const task = { id: state.nextId++, name, priority, status: 'todo' };
    state.tasks = [...state.tasks, task]; // FIX: Create a new array
    scheduleUpdate(() => emit('state:change', getSnapshot()));
}

export function moveTask(id, newStatus) {
    // FIX: Create a new array, mapping to update the specific task
    state.tasks = state.tasks.map(t =>
        t.id === id ? { ...t, status: newStatus } : t
    );
    scheduleUpdate(() => emit('state:change', getSnapshot()));
}

export function deleteTask(id) {
    // FIX: Create a new array, filtering out the deleted task
    state.tasks = state.tasks.filter(t => t.id !== id);
    scheduleUpdate(() => emit('state:change', getSnapshot()));
}

export function resetBoard() {
    state.tasks = [];
    state.nextId = 1;
    selectorCache.clear();
    scheduleUpdate(() => emit('state:change', getSnapshot()));
}

export function getSnapshot() {
    return {
        tasks: state.tasks,
        stats: {
            total: state.tasks.length,
            todo: state.tasks.filter(t => t.status === 'todo').length,
            doing: state.tasks.filter(t => t.status === 'doing').length,
            done: state.tasks.filter(t => t.status === 'done').length,
            highPriority: state.tasks.filter(t => t.priority === 'high').length,
        }
    };
}
```

Why This Works

This fix works by embracing the principle of immutability for the `state.tasks` array. Instead of modifying the existing `state.tasks` array in place (which keeps the same memory address), `addTask`, `moveTask`, and `deleteTask` now always create a *new array* when tasks are added, moved, or deleted. This means that after any modification, the `state.tasks` variable will point to a *different array object* in memory. Consequently, the `select` function's cache invalidation check (`cached.input === state.tasks`) will correctly evaluate to `false` because the references no longer match. This forces the `select` function to recompute the filtered task lists using the most up-to-date `state.tasks` data, which then gets passed to `renderBoard` and finally updates the UI correctly.
