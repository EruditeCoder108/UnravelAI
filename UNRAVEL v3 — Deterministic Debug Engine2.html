Concept To Learn

State Management & Caching

Immutability vs. Mutability in State Management

When building apps, especially with complex data, it's very easy to accidentally use old, stale data if you're not careful about *how* you change things. Using 'immutable updates' (creating new copies when changing data) makes it much clearer when something has actually been updated. This helps systems like caches or UI rendering libraries correctly know when to react and show fresh information, preventing confusing 'stale data' bugs that are often hard to find.

PATTERN TO AVOID:

Never directly modify (mutate) an array or object that is being used by a caching system or a UI framework's change detection mechanism. Instead, always create a *new* array or object with the desired changes.

💡 Imagine you have a special stamp for your important documents, and you use it to mark the *folder* (the array) where your files (tasks) are kept. If you add, remove, or change files *inside* the folder, but don't get a new folder or apply a new stamp to the *folder itself*, your office assistant (the `select` function) will think 'Oh, it's the same old folder with the same stamp, nothing new here!' and give you the old report, even though the files inside have changed. To get an updated report, you need to either give the assistant a *new* folder with a new stamp, or clearly tell them to re-check the *contents* of the old folder. In programming, creating a 'new folder' (a new array/object reference) is the cleaner way to signal a change.

Invariant Violations

  • The `state.tasks` array reference MUST change whenever tasks are added, removed, or their properties (like `status`, `name`, `priority`) are modified, for the `select` caching mechanism to function correctly.
  • The `selectorCache` MUST be cleared whenever a broad state reset occurs (e.g., `resetBoard`), or its entries must be explicitly updated or invalidated when relevant state changes happen.
  • The `scheduleUpdate` mechanism MUST ensure that `emit('state:change', getSnapshot())` is called for every state-modifying action (`addTask`, `moveTask`, `deleteTask`, `resetBoard`) to trigger UI updates.
  • The `getSnapshot()` function MUST always return a representation of the *current* `state.tasks` at the time it is called.
  • Event listeners (via `on`, `off`) should be managed correctly to prevent memory leaks and ensure `handleStateChange` is always active.

Technical Root Cause

The `select` function in `store.js` at line 56 uses reference equality (`cached.input === state.tasks`) to determine if its cached output is still valid. However, the functions `addTask`, `moveTask`, and `deleteTask` (lines 63, 73, 79 respectively) mutate the `state.tasks` array or its elements directly without changing the array's reference. This means that after an update, `select` incorrectly believes `state.tasks` is unchanged and returns stale data, preventing the UI from rendering the latest state.

Location: store.js line 56, in `select` function, specifically `cached.input === state.tasks`

Alternative Hypotheses

  • Stale Selector Cache after Task Mutation: The `select` function caches results based on the *reference equality* of `state.tasks`. When `addTask`, `moveTask`, or `deleteTask` modify `state.tasks` (or its elements) *in place*, the reference to `state.tasks` remains the same. This causes `select` to return stale cached data, leading to the UI not updating correctly for task columns or stats.
  • Event Emission Timing Issue: The `scheduleUpdate` uses `queueMicrotask`. If there's any synchronous UI rendering or state access happening *after* a mutation but *before* the microtask queue flushes, it might see a transiently inconsistent state. However, `getSnapshot` runs within the microtask and accesses the already mutated `state`, so this is less likely.
  • Incorrect DOM update logic in `renderColumn`: The `renderColumn` function replaces `el.innerHTML` entirely. If event listeners or specific DOM elements were not updating properly, this could cause problems. However, `onclick` attributes are global and re-created, so this is likely not the issue.

Why This Works

The fix works by ensuring that every time `state.tasks` is modified (when a task is added, moved, or deleted), a *new* array object is created and assigned to `state.tasks`. For `moveTask`, the task object itself is also recreated with the updated status. This means the `state.tasks` array's *reference* will change. When `select` is then called, its condition `cached.input === state.tasks` will correctly evaluate to `false` because `state.tasks` is now a different array object. This forces `select` to re-run the `selectorFn`, compute the `result` with the latest data, and update its cache with the new array reference, thus providing `renderBoard` with fresh, up-to-date task lists for rendering.
