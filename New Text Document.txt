The Problem With Specific Prompts
If the system prompt says "look for async boundaries, closure captures, state mutations" — you've already decided what matters before seeing the code.
What if the bug is a type coercion issue? "5" + 3 = "53". No async. No closure. No mutation. A narrowly written prompt would walk right past it.
What if the codebase is a pure functional pipeline with no state at all? The "map state mutations" instruction produces nothing useful and wastes the model's attention on something irrelevant.
What if the security vulnerability is a hardcoded API key on line 3? No OWASP pattern needed. No data flow analysis needed. It's just there. A prompt that says "trace user input through the system" would miss it entirely.
Narrow prompts produce narrow diagnoses. The model is obedient — it looks for what you tell it to look for and stops there.

What A Wide Prompt Actually Means
A wide prompt doesn't tell the model what kind of problem to find. It tells the model how to think — and then gets out of the way.
The difference:
Narrow: "Look for async boundaries, state mutations, closure captures, race conditions."
Wide: "Read this code completely. Understand what it is trying to do. Understand what it is actually doing. Find where those two things differ."
The second prompt works on async bugs, type coercion bugs, logic errors, off-by-one errors, wrong algorithm choices, missing edge cases — anything. Because it doesn't presuppose the category of problem.

The Right Mental Model For Each Mode
Debug Mode — The Right Question:
Not "what async/state/closure issues exist" but:

"What is this code trying to do? What is it actually doing? Where exactly do those diverge?"

That question is valid for every class of bug that has ever existed. The model then uses whatever lens is appropriate — async, state, type coercion, logic, data flow — because the code itself tells it which lens matters, not the prompt.
Explain Mode — The Right Question:
Not "describe the architecture, entry points, data flow" but:

"If a developer who has never seen this code needs to work on it tomorrow, what do they need to understand? What does this code do, how does it do it, and what would surprise them?"

That question produces useful explanations for a React app, a CLI tool, a backend API, a game engine, a data pipeline — anything. Because it's asking about the reader's needs, not presupposing the structure of the code.
Security Mode — The Right Question:
Not "check for OWASP Top 10, trace user input, look for missing auth" but:

"What assumptions does this code make about the people and systems that will interact with it? Where could a malicious actor violate those assumptions?"

That question catches XSS, SQL injection, hardcoded credentials, missing rate limiting, insecure direct object references, timing attacks — the full surface. Because it's asking about trust assumptions, which is what security actually is.
