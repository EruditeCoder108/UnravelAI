Concept To Learn
COPY
ASYNC_ORDERING, STATE_MUTATION
Managing Asynchronous State Updates and Race Conditions
In modern apps, data often comes from outside (APIs, web workers). If you don't handle these 'out-of-order' deliveries, your app can show old or wrong information, confusing users or leading to incorrect actions (like trying to build a unit that was just made unavailable). Unhandled errors can crash the app like a broken auto-rickshaw in traffic.

PATTERN TO AVOID:
Directly assigning the result of an asynchronous operation to a state variable in a UI component without checking if it's still the *most relevant* or *latest* data, especially when the operation is triggered frequently. Also, neglecting error handling (`.catch()`) for Promises.

💡 Imagine you're ordering sabzi (vegetables) from two different vendors via phone. First, you call Vendor A for potatoes. Then, a minute later, you call Vendor B for onions. Vendor B delivers very quickly. But then Vendor A, who was slow, delivers their potatoes *after* Vendor B. If you just put the last delivered sabzi in your basket, you might accidentally put old potatoes in after fresh onions, making your sabzi basket outdated. To avoid this, you'd need to put a 'latest order number' on your basket, and only accept deliveries that match your current latest order. For errors, imagine your auto-rickshaw breaks down; if you don't call the mechanic (`.catch()`), you'll just be stuck on the road!

Technical Root Cause
COPY
The `tick()` method in `UnitDisplay` (around line 74) makes asynchronous calls to `player.buildables()` without properly managing the order of resolutions or handling potential rejections. Since `tick()` is called frequently, multiple `buildables` promises can be pending. If a later request resolves *before* an earlier one, the earlier (stale) data will overwrite `this.playerBuildables`, leading to outdated UI. Additionally, the lack of a `.catch()` block means any promise rejection will go unhandled.

Alternative Hypotheses
Hypothesis 1 (Race Condition with Stale Data): The `tick()` method repeatedly calls `player.buildables()` which returns a Promise. Without a mechanism to cancel or ignore older, slower-resolving promises, an earlier request's resolution could overwrite `this.playerBuildables` with stale data if a later request resolves first.
Hypothesis 2 (Unhandled Promise Rejection): The `player.buildables().then(...)` call does not have a `.catch()` block. If the `player.buildables()` Promise rejects (e.g., due to a network error or internal processing error in the Web Worker), this rejection will be unhandled, leading to a console error and potential application instability.
Hypothesis 3 (No Practical Race Condition/Bug): The `player.buildables()` method is synchronous or so fast that race conditions rarely occur, and errors are not expected. This hypothesis was eliminated because the architecture (client -> `PlayerView` -> `WorkerClient` -> Web Worker) strongly suggests asynchronous behavior with variable delays, making race conditions and rejections highly probable.

Why This Works
The fix addresses the race condition by introducing `_lastBuildableRequestSeq`. Each call to `tick()` increments this sequence number and associates it with the new `player.buildables()` promise. When a promise resolves, its `.then()` callback checks if its `currentRequestSeq` matches the `_lastBuildableRequestSeq`. If they match, it means that this is the latest request, and its data can safely update `this.playerBuildables`. If they don't match, it means a newer request has already been made, and this resolving promise's data is now stale and should be ignored, preventing the overwrite. The `.catch()` block ensures that any errors from `player.buildables()` are caught and handled, preventing unhandled promise rejections, and reflecting the error state in the UI if it's the latest request to fail. Separate `this.requestUpdate()` calls for synchronous changes ensure the UI reflects all state changes appropriately.
